1. 目的・背景
現状の課題本施策での解決
過去ラウンドはURLを管理しないとアクセスできないログインすれば自動的に履歴が残る
個人スタッツ（平均スコア・ベスト等）が見られないマイページで通算成績・ホール別傾向を確認できる
パット数が記録できないショット数と別にパット数を入力・分析できる
メンバーが誰かを特定する仕組みがない任意ログインで自己紐づけが可能になる

📌 メンバー登録のポリシー（変更なし）
メンバー登録は引き続き「誰でも自由」。幹事がメンバー登録しても、招待URLを受けたメンバーが自分で登録することも可能。ログインは義務ではなく、ゲストとして今まで通り使い続けることもできる。

4. データモデル（確定版）
// ── 既存（変更なし） ───────────────────────────────────────────
sessions/{sessionId}/
  members:  [...]
  scores:   { "田中_0": 4, "田中_1": 5, ... }
  holes:    9 | 18
  pars:     [4, 4, 4, ...]

// ── 既存 + フィールド追加 ────────────────────────────────────────
rounds/{sessionId}/
  // 既存フィールド
  createdAt:   timestamp
  holes:       9 | 18
  label:       string
  date:        "YYYY-MM-DD"
  confirmedAt: timestamp | null
  // 新規追加
  createdBy:   uid | null         // ログイン幹事が作成時にセット
  linkedPlayers:              // 「この◯◯は自分」と紐づけ時にセット
    {uid}: "田中"

// ── 新規追加 ─────────────────────────────────────────────────────
sessions/{sessionId}/
  putts:                           // パット数（Phase 6）
    "田中_0": 2
    "田中_1": 1

users/{uid}/                        // ユーザープロフィール
  displayName: "田中太郎"
  email:       "..."
  createdAt:   timestamp
  agreedToPolicy: timestamp      // PP同意日時

userRounds/{uid}/               // 高速検索用インデックス
  {sessionId}: true              // 作成時 or 自己紐づけ時に書き込み
⚠️ なぜ userRounds インデックスが必要か
Firebase Realtime Database はSQLのWHERE句が使えないため、「createdBy=uid OR linkedPlayers に uid がある rounds」を1クエリで取得できない。userRounds/{uid}/{sessionId}=true を別途管理することで、自分のラウンドID一覧をO(1)で取得可能になる。

主要フロー一覧
フロー説明
幹事がラウンドを作成 ログイン済みで「新しいラウンド」→ createdBy=uid 保存 → userRounds に自動追加 → URLをシェア（今まで通り）
ゲストがURLを開く 変更なし。ログイン不要でフル利用可能
ログインユーザーがURLを開く メンバー画面に自己紐づけUIが表示される。選択すると userRounds に追加
マイページを開く ログイン済み → 履歴＆スタッツ表示。未ログイン → ログイン促進画面
アカウント削除 確認後 users/{uid} / userRounds/{uid} / linkedPlayers内のuid を削除 → Firebase Auth アカウントも削除
6. 実装フェーズ
0
Phase 0｜プライバシーポリシー
小
必須・最優先
メールアドレス収集前に法的対応を完了する。実装Phase 1より先に着手すること。
  • プライバシーポリシーページの作成（収集情報・利用目的・Firebaseへの保存・越境移転・削除方法を明記）
  • アプリフッターにプライバシーポリシーへのリンクを設置
  • メール登録画面に「利用規約・プライバシーポリシーに同意する」チェックボックスを追加
  • 同意日時を users/{uid}/agreedToPolicy に保存
前提：なし（単独で着手可能）
1
Phase 1｜認証基盤
中
必須
Firebase Auth の有効化とアプリへの組み込み。
  • Firebase Auth インポート追加（Google + Email/Password）
  • ヘッダーにログイン状態表示（ユーザー名 or ログインボタン）
  • ログイン・ログアウト・新規登録・パスワードリセットモーダルの実装
  • メール確認メール送信（sendEmailVerification）
  • users/{uid} へのプロフィール保存
  • Firebase Security Rules の再設計（rounds全件読み取り廃止）
前提：Firebase Console セットアップ完了（Section 8参照）、Phase 0完了
2
Phase 2｜ラウンドへの UID 紐づけ
小
ログイン中に新規ラウンドを作成した場合に createdBy を自動記録。
  • initSession() / newSession() でログイン中なら createdBy=uid 保存
  • userRounds/{uid}/{sessionId}=true を書き込み
前提：Phase 1完了
3
Phase 3｜自己紐づけ UI（メンバー画面）
中
他人が作成したラウンドに自分の参加記録を紐づける。
  • メンバー画面に「このラウンドの自分を選択」カードを追加（ログイン時のみ表示）
  • linkedPlayers/{uid} と userRounds/{uid} への書き込み
  • 紐づけ済み表示・変更ボタン・取消ボタン
  • 他UIDが選択済みのメンバーをグレーアウト
前提：Phase 1・2完了
4
Phase 4｜マイページ（ラウンド履歴）
大
④マイページタブを追加し、自分のラウンド一覧を表示。
  • ④マイページタブ追加・画面実装
  • 未ログイン時のログイン促進UI
  • userRounds インデックスからラウンド一覧を取得・表示
  • 作成🏌 ／ 参加👤 バッジ、日付・ラベル・スコア・PAR差を表示
  • 「スコアを開く」でそのラウンドのスコア画面に遷移
前提：Phase 2・3完了
5
Phase 5｜マイページ（個人スタッツ）
大
全ラウンドを集計してホール別スタッツまで表示。
  • 個人スタッツ集計（ラウンド数・平均スコア・ベスト・平均PAR差）
  • ホール別スタッツ（平均スコア・バーディー率・ボギー率・ダボ以上率）
  • アカウント管理UI（表示名変更・メール確認再送・ログアウト・アカウント削除）
  • アカウント削除時のデータクリーンアップ処理
前提：Phase 4完了
6
Phase 6｜パット数入力・スタッツ
中
ショット数に加えてパット数を記録・分析する。
  • カウント画面にパット数入力UI追加（0〜9、任意）
  • sessions/{sessionId}/putts のFirebase保存・リアルタイム同期
  • スコア画面にパット数列追加（未入力は「−」）
  • マイページに平均パット数・ホール別パットスタッツ追加
  • CSVダウンロードにパット数列を追加
前提：Phase 5完了（パットスタッツ部分のみ。入力UI自体は独立して実装可能）
7. セキュリティ・プライバシー
✅ Firebase Auth が担保する部分（対応不要）
パスワードのハッシュ化保管 / Google OAuth 2.0準拠 / JWT トークン自動管理 / HTTPS通信 / Netlify 自動HTTPS
🚨 要対応：Firebase Security Rules の修正（Phase 1で実施）

現状の rounds/.read: true は全件取得を許可しており、認証機能追加後に linkedPlayers: {uid: "田中"} が誰でも閲覧可能になる問題がある。

修正後は rounds/$sessionId/.read: true（親レベルは削除）に変更し、管理者画面の取得ロジックを合わせて修正する。

⚠️ 要注意：管理者パスワードの平文記載（将来課題）
現在 const ADMIN_PASSWORD = 'NuuGolf' がHTMLソースに記載されており、個人情報取り扱い後は重要度が上がる。本施策完了後に管理者もFirebase Authで認証する方式への移行を推奨。

対応項目必要度内容

利用目的の明示 必須 登録画面に「ゴルフスコア管理目的」と明記

8. Firebase Console セットアップ手順
Phase 1 実装前に完了してください。Firebase Console は console.firebase.google.com から操作します。
STEP 1｜Authentication を有効化
  1. Firebase Console を開き、プロジェクトを選択する
  2. 左メニュー「構築」→「Authentication」をクリック
  3. 「始める」ボタンをクリック
STEP 2｜Google ログインを有効化
  1. 「Sign-in method」タブをクリック
  2. 「Google」をクリック
  3. 右上のトグルを「有効」にする
  4. 「プロジェクトのサポートメール」を自分のメールに設定
  5. 「保存」をクリック
STEP 3｜メール／パスワードを有効化
  1. 「メール/パスワード」をクリック
  2. 上段「メール/パスワード」トグルを「有効」にする
    下段の「メールリンク（パスワードなしのログイン）」はOFFのまま
  3. 「保存」をクリック
STEP 4｜承認済みドメインを追加
  1. 「Sign-in method」タブ下部「承認済みドメイン」を確認
  2. 「ドメインを追加」をクリック
  3. Netlify のドメインを入力して追加
    例：our-golf-score-check.netlify.app　※ localhost は登録済みのため不要
STEP 5｜Realtime Database セキュリティルール更新
  1. 左メニュー「Realtime Database」→「ルール」タブをクリック
  2. 既存ルールを以下に置き換える
{
  "rules": {
    "sessions": {
      "$sessionId": {
        ".read": true,
        ".write": true
      }
    },
    "rounds": {
      "$sessionId": {
        ".read": true,   // ← 親レベルの .read:true を削除（全件列挙防止）
        ".write": true
      }
    },
    "courses": {
      ".read": true,
      ".write": true
    },
    "users": {
      "$uid": {
        ".read":  "auth !== null && auth.uid === $uid",
        ".write": "auth !== null && auth.uid === $uid"
      }
    },
    "userRounds": {
      "$uid": {
        ".read":  "auth !== null && auth.uid === $uid",
        ".write": "auth !== null && auth.uid === $uid"
      }
    }
  }
}
  1. 「公開」をクリック
⚠️ このルール変更で管理者画面の全ラウンド取得が制限される
現状の管理者画面は rounds/ 全件取得に依存しているため、Phase 1 の実装時に管理者の取得ロジックを合わせて修正すること。

📌 総合評価
このアプリは友人グループ内のゴルフスコア管理が目的であり、金融・健康・位置情報などのセンシティブデータは扱わない。適切にPhase 0（PP対応）とPhase 1（Rules修正）を実施すれば、残存リスクは低く抑えられる。
